# Writing Skill

Professional writing methodology for crafting clear, compelling content across formats.

## When to Use This Skill

Use this skill when the user asks to:
- Write an email, message, or letter
- Draft a blog post or article
- Create documentation or guides
- Write reports or summaries
- Compose social media content
- Draft any written communication

## Core Principle: Write for the Reader

Every piece of writing serves a reader. Before writing:
1. **Who** is the audience? (expertise, interests, concerns)
2. **Why** are they reading? (inform, persuade, entertain, instruct)
3. **What** do they need to take away?
4. **How** much time do they have?

## The Writing Process

### Step 1: Understand the Brief

Before writing, clarify:

| Question | Purpose |
|----------|---------|
| **Goal** | What should the reader think/feel/do after reading? |
| **Audience** | Who is this for? What do they already know? |
| **Tone** | Formal, casual, technical, friendly, authoritative? |
| **Format** | Email, article, report, social post, documentation? |
| **Length** | Word count, sections, level of depth? |
| **Constraints** | Platform limits, style guides, templates? |

### Step 2: Research (if needed)

For factual content:
- Gather current information
- Find examples and case studies
- Note statistics and quotes
- Consider counter-arguments

### Step 3: Structure First

**Always outline before writing.** Structure depends on format:

#### For Articles/Blog Posts
```
# [Compelling Title]
## Hook (attention-grabbing opening)
## Context (why this matters)
## Main Points
- Point 1: [key idea + evidence]
- Point 2: [key idea + evidence]
- Point 3: [key idea + evidence]
## Conclusion (takeaway + call to action)
```

#### For Emails
```
Subject: [Clear, actionable subject]
[Greeting]
[Purpose in first sentence]
[Key details - bullet points if multiple]
[Clear ask or next step]
[Sign-off]
```

#### For Reports
```
# Executive Summary (key findings in 2-3 sentences)
## Background
## Methodology
## Findings
## Recommendations
## Appendix (detailed data)
```

#### For Documentation
```
# [Feature/Concept Name]
## Overview (what and why)
## Quick Start (fastest path to success)
## Detailed Guide (step-by-step)
## Reference (options, parameters, examples)
## Troubleshooting
```

### Step 4: Write the First Draft

Focus on getting ideas down. Don't edit while writing.

**Opening strategies:**
- **Question**: "Ever wondered why...?"
- **Statistic**: "80% of users abandon..."
- **Story**: "Last week, our team faced..."
- **Bold claim**: "Everything you know about X is wrong."
- **Problem statement**: "If you've struggled with..."

**Body strategies:**
- One idea per paragraph
- Lead with the main point, then support
- Use transitions: "However," "Additionally," "As a result,"
- Include specific examples
- Use analogies for complex concepts

**Closing strategies:**
- Summarize key takeaways
- Call to action (what should reader do?)
- Forward-looking statement
- Circle back to opening

### Step 5: Polish

Apply these refinements:

| Issue | Fix |
|-------|-----|
| Passive voice | "The report was written" → "We wrote the report" |
| Wordiness | "In order to" → "To" |
| Weak verbs | "made a decision" → "decided" |
| Nominalizations | "give consideration to" → "consider" |
| Filler words | Remove "very," "really," "just," "actually" |
| Long sentences | Break into shorter ones |
| Redundancy | "past history" → "history" |

## Writing Style by Format

### Professional/Business Email

```
**Subject:** Clear action + context

Hi [Name],

[Purpose in one sentence.]

[Key details:]
- Point 1
- Point 2
- Point 3

[Clear next step: "Could you review by Friday?" or "Let me know your thoughts."]

Best,
[Name]
```

**Email principles:**
- Subject line tells them what and why
- First sentence = purpose (no "I hope this email finds you well")
- Bullet points for multiple items
- One clear ask per email
- Mobile-friendly (short paragraphs)

### Technical Documentation

```markdown
# Feature Name

Brief description of what this does.

## Quick Start

\`\`\`bash
# Fastest way to get started
npm install package
\`\`\`

## Usage

### Basic Example
\`\`\`typescript
import { feature } from 'package';
feature.doSomething();
\`\`\`

### Advanced Example
\`\`\`typescript
// More complex use case with comments
\`\`\`

## API Reference

### `functionName(params)`
- **params.option1** (string): Description
- **params.option2** (number, optional): Description. Default: `10`
- **Returns**: Description of return value

## Common Issues

### Error: "Something went wrong"
This happens when... Fix by...
```

**Documentation principles:**
- Lead with what they need most (Quick Start)
- Show, don't tell (code examples first)
- Be precise (exact function names, types)
- Cover errors they'll encounter

### Blog Post/Article

```markdown
# [Compelling Title with Benefit]

[Hook: Story, question, or bold statement]

[Context: Why this matters to the reader]

## [First Main Point]

[Lead with the point]

[Supporting evidence/example]

[Implication for reader]

## [Second Main Point]

...

## [Third Main Point]

...

## Conclusion

[Key takeaway]

[Call to action]
```

**Article principles:**
- Title promises value
- Hook in first 50 words
- Subheadings as mini-titles (scannable)
- One idea per section
- End with clear next step

### Social Media

| Platform | Best Practices |
|----------|----------------|
| **Twitter/X** | Hook in first line, thread for length, end with CTA |
| **LinkedIn** | Professional tone, use line breaks, ask for engagement |
| **Slack/Discord** | Scannable, use threads, respect async |

## Tone Calibration

### Formal
- Complete sentences
- No contractions
- Third person
- Precise terminology
- "The analysis indicates..."

### Professional
- Complete sentences
- Some contractions okay
- First/second person
- Industry terms
- "We found that..."

### Casual
- Shorter sentences
- Contractions natural
- Direct address ("you")
- Plain language
- "Here's what we learned..."

### Friendly
- Conversational
- Personal touches
- Humor where appropriate
- "Hey, guess what we discovered?"

## Common Mistakes to Avoid

| Mistake | Problem | Fix |
|---------|---------|-----|
| Buried lede | Key point hidden | Lead with main message |
| Wall of text | Unreadable | Break into paragraphs, use headers |
| Jargon overload | Excludes readers | Define terms or use plain language |
| Passive voice | Weak, unclear | Use active voice |
| No clear ask | Reader confused | End with specific next step |
| Too long | Won't be read | Cut 20%, then cut 20% more |
| Hedging | Undermines message | "I think maybe" → "Consider" |

## Checklist Before Sending

### Structure
- [ ] Clear purpose in first paragraph
- [ ] Logical flow between sections
- [ ] Key points easy to find (scannable)
- [ ] Clear conclusion/call to action

### Clarity
- [ ] One idea per paragraph
- [ ] Active voice preferred
- [ ] Jargon explained or avoided
- [ ] Specific examples included

### Polish
- [ ] Filler words removed
- [ ] Sentences varied in length
- [ ] Read aloud for rhythm
- [ ] Proofread for typos

### Audience
- [ ] Appropriate tone
- [ ] Assumed knowledge level correct
- [ ] Format matches context
- [ ] Length respects reader's time
